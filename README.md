# Карта знаний R&D — горно-металлургическая отрасль

Единая **карта знаний (knowledge graph)** для НИОКР в горной металлургии: связывает
публикации, эксперименты, технологические решения, материалы, оборудование, экспертов
и выводы, и отвечает на сложные многопараметрические запросы на естественном языке
(RU/EN) с учётом числовых диапазонов, географии и уровня достоверности.

Решение самодостаточно: работает на чистом Python + Flask, без внешних БД и без доступа
в интернет. Граф хранится в `data/graph.json`; реальные данные подключаются модулем импорта.

---

## Что умеет (соответствие ТЗ)

| Требование ТЗ | Реализация |
|---|---|
| Онтология домена (Material, Process, Equipment, Property, Experiment, Publication, Expert, Facility, Finding, Domain) | `kg/ontology.py` |
| Отношения (uses_material, operates_at, produces_output, described_in, validated_by, contradicts, …) | `kg/ontology.py`, валидируются движком |
| NLP-извлечение сущностей, синонимы RU/EN, сокращения (ПВП/flash smelting, электроэкстракция/electrowinning) | `kg/normalize.py` |
| Числовые ограничения и диапазоны (сульфаты ≤300 мг/л, 200–300 мг/л, от 100 т/сут) с приведением единиц | `kg/normalize.py` |
| Хранение графа, обход связей на 3–4 уровня | `kg/engine.py` (networkx) |
| Версионирование фактов + ручная корректировка экспертами | `changelog`, `annotate()` |
| Семантико-структурный поиск: материал + процесс + условие + география + период + числа | `kg/query.py` |
| Литобзор, консенсус/разногласия, число подтверждающих источников | `kg/synthesize.py` |
| Выявление пробелов («холодный климат + кучное выщелачивание + никель») | `synthesize.gap_analysis()` |
| Сравнение технологий по параметрам (эффективность, CAPEX/OPEX, холодный климат) | `synthesize.comparison_table()` |
| Дашборды: покрытие по доменам, активность команд, зоны риска | `synthesize.dashboards()` |
| Визуализация графа: цепочки «материал → процесс → оборудование → результат», подсветка противоречий | фронтенд `static/index.html` |
| Различение отечественной / зарубежной / мировой практики | поле `geography`, фильтр в поиске |
| Верификация знаний: источник, достоверность, дата актуализации | провенанс на каждом узле/ребре |
| Экспорт PDF / Markdown / JSON-LD (FAIR) | `kg/export.py` |
| Мультиязычность RU/EN | нормализация + `label`/`label_en` |
| Импорт разнородных данных (CSV/JSON/текст) | `scripts/ingest.py` |

Все **4 эталонных запроса** из ТЗ отрабатываются из коробки (см. чипы-примеры в интерфейсе).

## Быстрый старт

```bash
pip install -r requirements.txt
python app.py
# открыть http://127.0.0.1:8000
```

При первом запуске граф строится из доменного датасета (`kg/seed.py`) и сохраняется
в `data/graph.json`.

### Подключение реальных данных

Положите файлы в `data/import/` и выполните импорт:

```bash
python -m scripts.ingest        # поверх seed-графа
python -m scripts.ingest --fresh  # только импортируемые данные
```

Поддерживаются:
- `*.csv` — каталог экспериментов (колонки: `id, название, материал, режим, результат, год, география, достоверность, значение, единица, параметр`);
- `*.json` — публикации/отчёты (`id, title, year, geography, domain, authors, abstract, source`);
- `*.txt` / `*.md` — свободные отчёты (авто-извлечение терминов и числовых условий).

В репозитории лежат образцы (`experiments_sample.csv`, `publications_sample.json`,
`report_sample.txt`) — их можно заменить своими.

> Данные кейса (Я.Диск) в среде сборки были недоступны, поэтому включён репрезентативный
> доменный датасет со структурой 1-в-1 под описанные входные данные. Реальный корпус
> подключается указанным выше импортом без изменений кода.

## Архитектура

```
mining-kg/
├── app.py                 Flask: REST API + раздача фронтенда
├── kg/
│   ├── ontology.py        типы сущностей/отношений, домены, единицы, достоверность
│   ├── normalize.py       синонимы RU/EN, стемминг, парсинг чисел и диапазонов
│   ├── engine.py          граф (networkx): узлы, рёбра, провенанс, версии, обход
│   ├── query.py           разбор NL-запроса → фильтры → ранжирование
│   ├── synthesize.py      литобзор, консенсус, пробелы, сравнение, дашборды
│   ├── export.py          Markdown / JSON-LD / PDF
│   └── seed.py            доменный датасет (покрывает 4 эталонных запроса)
├── scripts/ingest.py      импорт CSV/JSON/текста
├── static/index.html      SPA: поиск, синтез ответа, интерактивный граф, дашборды
└── data/                  graph.json + import/
```

### Поток запроса
`NL-запрос → parse_query (термины+гео+период+числа) → обход графа + ранжирование →
синтез (обзор/консенсус/пробелы/сравнение/эксперты) → ответ + подсветка окрестности графа`.

### Технологии
- **Граф**: networkx `MultiDiGraph` (замена Neo4j без внешней БД; при необходимости
  граф сериализуется в JSON и переносится в Neo4j/Neptune — схема совместима с онтологией).
- **NLP**: собственный лёгкий пайплайн (словарь синонимов + стемминг + regex-парсер
  числовых ограничений). Точки расширения под spaCy/ruBERT отмечены в коде.
- **Онтология**: типизированные узлы/рёбра с валидацией (аналог OWL/SHACL-ограничений).
- **FAIR**: экспорт JSON-LD с контекстом schema.org.

## API (кратко)

| Метод | Назначение |
|---|---|
| `POST /api/answer {query}` | полный синтезированный ответ |
| `POST /api/search {query,limit}` | ранжированные результаты |
| `GET /api/graph?node=&depth=` | подграф-окрестность для визуализации |
| `GET /api/node/<id>` | карточка сущности со связями |
| `GET /api/dashboards` | метрики покрытия, активность, риски, пробелы |
| `POST /api/compare {ids}` | таблица сравнения |
| `POST /api/annotate {id,note,author}` | экспертная правка графа |
| `POST /api/export {query,format}` | md / pdf / jsonld |

## Ограничения и развитие
- Текущий NLP — словарно-эвристический; для промышленного объёма подключаются
  ruBERT/DeepPavlov (извлечение сущностей и отношений) и spaCy-пайплайн.
- Для >1 млн сущностей граф переносится в Neo4j/JanusGraph (Cypher/Gremlin); текущая
  онтология и сериализация к этому готовы.
- Семантический поиск можно усилить эмбеддингами + Elasticsearch/Vespa для гибрида
  «вектор + структурный фильтр».
